## A. 注文番号・識別子

| # | シナリオ | 現仕様 | 判定 | 対処方針 |
|---|---|---|---|---|
| A-1 | 同じ注文番号が同日に 2 人へ発行される（同時注文で連番が競合） | 採番方式未定義（レビュー L-1） | **新** | 店舗単位の DynamoDB アトミックカウンタで採番。日次リセットするなら「日付 + 番号」を内部キーにする |
| A-2 | 昨日の未受取 #042 と今日の #042 が同時に READY_PICKUP に並ぶ | 定義なし | **新** | 営業終了時に残注文を一括クローズ（→ D-5）+ スタッフ画面は番号だけでなく明細・時刻を併記して照合 |
| A-3 | 受け渡し時、番号の見間違い・言い間違いで別人の注文を渡す | 照合手順なし | **運** | 受け渡しスワイプ前に「番号 + 商品内容」の 2 点照合を運用ルール化 |

## B. 通信エラー（クライアント⇄サーバ）

| # | シナリオ | 現仕様 | 判定 | 対処方針 |
|---|---|---|---|---|
| B-1 | `POST /orders` がタイムアウト。注文が作られたか客に分からない | `Idempotency-Key` あり（requirements §4.3） | **済** | 同一キー再送で同じ Order が返る設計は正しい。ただし**クライアント側の再送手順**（キーは注文確認画面の表示時に生成・リトライは同一キー・N 回失敗で「店頭でご注文ください」表示）を要件に明文化する |
| B-2 | 201 は返ったがレスポンス受信前に接続断。客は orderId / 注文番号を知らないまま | **注文を再発見する API がない** | **新（High 相当）** | `GET /orders?sessionId=` （自分の当日注文一覧）を追加。アクセスパターン `SESSION#<sessionId>` → 注文、の GSI が必要。B-4・C-1 の回復経路も兼ねる |
| B-3 | WebSocket が無音で死ぬ（Wi-Fi→4G 切替、キャリア NAT タイムアウト）。visibilitychange は発火せず画面が古いまま | フォアグラウンド中の再取得手段はキュー位置ポーリング（30 秒）のみ。ステータスは WS 頼み（requirements §9） | **新** | ステータスも定期リコンサイル対象にする: 30 秒ポーリングのレスポンスに status を同梱（queue-position と統合）+ WS の onclose/onerror で再接続。「WS は高速化、ポーリングは保証」と役割を明記 |
| B-4 | ブラウザ再起動・localStorage 消去で sessionId / orderId を喪失 | sessionId はローカル保持のみ（§13） | **新** | B-2 の API でも救えない（sessionId ごと消えるため）→ 最終フォールバックは C-1 と同じ運用対応。注文確認画面に「スクリーンショット推奨」を出すのが安価な緩和策 |
| B-5 | スタッフのスワイプ確定時（3 秒後）にタブレットがオフライン → 書き込みが消える | アンドゥバナーは 3 秒間クライアント保持（requirements §5.4）。失敗時挙動は未定義 | **新** | 書き込み失敗時はカードを元状態に戻してエラー表示 + 自動リトライ。「バナー表示中の操作はまだサーバに存在しない」ことを設計上の前提として明記 |
| B-6 | 顧客キャンセルとスタッフの製造開始スワイプが同時発生 | 条件付き書き込み `fromStatus` で片方が 409（architecture §6.5） | **済** | 対処済み。409 を受けたスタッフ画面が「この注文はキャンセルされました」と表示してカードを消す、の UI 挙動だけ追記 |

## C. 手動フォールバック（人間系）

| # | シナリオ | 現仕様 | 判定 | 対処方針 |
|---|---|---|---|---|
| C-1 | 受け取り時に客のスマホが充電切れ。注文番号を提示できない | 照合手段はスマホ画面のみ。**注文に氏名等の人間可読な識別子が一切ない** | **新 + 運** | 受け渡し照合がスマホ画面という単一障害点に乗っている。対処: スタッフが READY_PICKUP 一覧から「注文内容 + おおよその注文時刻」の口頭照合で特定 → 受渡スワイプ、を運用ルール化。仕様側は注文時に任意ニックネーム（1 項目）を足すと照合が桁違いに楽になる |
| C-2 | 客が口頭で「やっぱりホットに変えて」 | 変更 API なし。キャンセルは STORE_ACCEPTED のみ | **運** | 変更 API は作らない（MVP 正解）。運用: STORE_ACCEPTED 中ならスタッフがキャンセル（レビュー H-3 が前提）→ 客が再注文。PREPARING 以降は店頭判断 |
| C-3 | マシン故障（例: Zone A の L 専用機が停止） | 再ルーティング・ゾーン停止の手段なし | **新** | ゾーン単位の受付停止フラグ（レビュー H-4 の店舗フラグをゾーン粒度に拡張）。滞留注文はスタッフキャンセル（H-3）で返す。自動再ルーティングは作らない（L を他機で作れない物理制約があるため、そもそも成立しない） |
| C-4 | バリスタ 1 人運用の時間帯に 4 ゾーンへ注文が分散 | 想定記述なし | **運** | システム対応不要。1 人が全ゾーンのタブレットを見る or 使わないゾーンを C-3 のフラグで閉める |
| C-5 | 店頭払いの客が支払わずに商品だけ受け取る／支払ったか分からない | 支払い状態はシステム外（§2.2） | **運** | 「受渡スワイプはレジ精算後に行う」を運用ルール化（スワイプ = 金銭授受完了の記録と定義する）。運用フロー 1 枚の必須項目 |

## D. サーバ・インフラ障害

| # | シナリオ | 現仕様 | 判定 | 対処方針 |
|---|---|---|---|---|
| D-1 | 注文は書けたが後段パイプライン（Streams→EventBridge→router→SQS→consumer）のどこかで失敗。zone が永遠に確定せず、スタッフに一切表示されない | リトライ・DLQ・監視の記述なし | **新（High 相当）** | ①EventBridge ターゲットと SQS に **DLQ** を設定 + CloudWatch アラーム ②突合バッチ: 「STORE_ACCEPTED かつ zone 未確定が N 分超」の注文を検出してイベント再発行（EventBridge Scheduler の最初の実用途になる）。**客は注文成功と信じて店に向かっているのに店側は存在を知らない**、という本システム最悪の障害モードなので優先度高 |
| D-2 | push-fn 障害で状態変化の通知が届かない | ポーリング併用（§4.4）だが B-3 の通りステータスはWS 単独 | **既（B-3 と同根）** | B-3 の「ポーリングに status 同梱」で自動的に緩和される。push は高速化手段、ポーリングが保証、の原則で統一 |
| D-3 | SQS の再配信で zone-consumer が同一メッセージを二重処理（visibility timeout 超過時。FIFO dedup は送信側 5 分窓の重複排除であり、消費側の再配信は防がない） | 冪等性の記述なし | **新** | queueSeq 書き込みを条件付き（zone 未設定時のみ）にして冪等化。1 行で済むが明記しないと事故る |
| D-4 | 全面障害（リージョン障害・デプロイ事故）。QR を読んでもページが開かない | 記述なし | **運** | 個人開発 MVP で技術対策（マルチリージョン等）は過剰。運用: 開かなければ客は自然に店頭注文に戻る。店側は「アプリ注文が来ない」ことに気づける必要があるので、D-1 のアラーム通知先（オーナーのメール等）だけ確保する |
| D-5 | 営業終了時に STORE_ACCEPTED / READY_PICKUP の注文が残る | 記述なし | **新** | クローズ処理を定義: 閉店時にスタッフが残注文を一括キャンセル（H-3 の遷移を流用）。将来は EventBridge Scheduler で 18:00 自動化（requirements §14.7 に既出の構想と接続） |

## 優先度の整理（仕様 v1.2 に入れるべき「新」項目）

| 優先 | 項目 | 理由 |
|---|---|---|
| 1 | D-1 パイプライン取りこぼし対策（DLQ + 突合） | 「客が来るのに店が知らない」は本システム最悪の障害モード |
| 2 | B-2 注文再発見 API（sessionId → 注文） | 通信断・画面喪失系の回復経路がすべてここに合流する |
| 3 | B-3 ステータスのポーリング保証（WS は高速化と再定義） | 通知系の障害（B-3/D-2）を一括で許容可能にする |
| 4 | A-1 採番仕様の確定 + B-5/B-6 のスタッフ UI 失敗時挙動 | 実装 Phase 4〜5 の直接入力 |
| 5 | C-3 ゾーン単位受付停止 + D-5 クローズ処理 | H-3（スタッフキャンセル）・H-4（受付停止）の具体化として同時に設計できる |

「運」判定 5 件（A-3 / C-2 / C-4 / C-5 / D-4）はコードではなく**店舗運用フロー 1 枚**（次アクション #4）の記載項目。C-5（支払い突合）は最重要。

## 冗長性の設計原則

本カタログの対処方針は、一貫して次の原則で選んでいる。

- **真実は 1 箇所、経路は 2 本以上**。注文の状態の真実は DynamoDB だけに置く（真実を複製しない — 複製は不整合の温床）。ただしそこへ到達する経路は必ず複線化する: WS が死んでもポーリングで届く（B-3）、イベントが落ちても突合バッチで拾う（D-1）、デジタルが全滅しても店頭運用で回る（C-1/C-5/D-4）
- **劣化の階段を設計する**。障害時に「全停止」へ落ちるのではなく、1 段ずつ落ちる: WS 即時通知 → 30 秒ポーリング → 客がスタッフに声をかける → 店頭注文。各段が下の段に自然に着地することを確認してから実装する
- **冗長化するのは流れであってインフラではない**（この規模では）。AZ 冗長はマネージドサービスが既に持っている（requirements §10.1）。マルチリージョン等のインフラ冗長は月額数百円のシステムには過剰。効くのは上記のような**経路の冗長化**で、ほぼタダで手に入る

## 教訓 — 異常系の列挙方法

異常系は「通信・時間・人間・機械」の 4 軸で正常系シーケンスの**矢印 1 本ずつ**に「これが失敗したら/遅れたら/二重に届いたら」を問うと機械的に列挙できる。今回の D-1 も、architecture.md §9 のシーケンス図の `D-->>EB` の矢印に「失敗したら？」を当てただけで見つかる。シーケンス図は描いて終わりではなく、異常系レビューの入力として使う。
