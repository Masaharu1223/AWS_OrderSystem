# Fable 思考プロセス記録 — 仕様書レビューをどう進めたか

**対象タスク**: カフェ向けモバイルオーダー仕様書（requirements.md / architecture.md）のレビュー + 次アクション整理
**目的**: 動作テストとして、入力の受け取りから結論までの思考の流れを開示する
**作成日**: 2026-07-06
**増補**: 同日、セッション全体（異常系展開・上流層の修正・スコープ挙動の観測）を §8〜§12 に追記。§1〜§7 は初回レビュー時点の記録のまま残す

## 1. 入力の分解

依頼文から 3 つの成果物を要求されていると解釈した。

1. 2 文書のレビュー（実装者へ手渡す前提）
2. 次の作業指示 + 要件定義段階で必要な他文書の案内
3. この思考プロセス自体の記録（本書）

同時に、環境ルールから自動適用される手続きを 2 つ判定した。

- **思考の深さ判定**: 設計レビュー + 複数成果物 = Red 寄り（「深く検討します」と宣言してから着手）
- **提供ファイルの整理**: パス提供されたファイルは元の場所（Downloads）に残さない → `~/.claude/inbox/cafe-mobile-order/` へ移動。ただし `requirements.md` / `architecture.md` は文書間で相互参照している標準名なので、日付リネームの例外（プロジェクト標準名）を適用して名前は維持した

## 2. レビュー戦略の設計

「上から読んで気になった所を挙げる」方式は網羅性が出ないため、4 本の突合軸を先に決めた。

| 軸 | 何を見るか |
|---|---|
| 文書内整合 | 同一文書内で矛盾する記述はないか |
| 文書間整合 | requirements の要求を architecture が実際に満たしているか（要求→実現手段のトレース） |
| アクター体験ウォークスルー | 顧客・バリスタ・オーナー・開発者本人の視点で、具体的なシナリオを最初から最後まで通す |
| 実装可能性 | AWS の実仕様・DynamoDB のクエリ制約に照らして、書いてある通りに作れるか |

このうち最も収穫が大きかったのは **アクター体験ウォークスルー**。技術構成の美しさ（この設計書はかなり美しい）は、体験の穴を隠す方向に働くため、意図的に「具体的な客」を想像して流した。

## 3. 主要発見の導出過程（代表 3 件）

### C-1（明細×ゾーンの矛盾）をどう見つけたか

1. §7.1 のゾーン定義を読んだ時点で「ゾーンはサイズで分かれている」ことに着目
2. 「では複数商品の注文はどうなるか」と自問 → architecture §8.2 に「**明細ごとに**ゾーン判定」とある
3. 具体注文でウォークスルー: 「カフェラテ L + 紅茶 M」→ Zone A と Zone D に分裂
4. この分裂した注文を、注文単位の仕様たち（ステータス遷移 §6 / QueuePosition の単一 `zone` / スタッフのカード操作 §5.3）に順に通す → 3 箇所すべてで挙動が定義不能になることを確認
5. 1 箇所の誤記ではなく「製造単位」という概念の欠落と結論 → Critical 判定

### H-1（新規注文 push の欠落）をどう見つけたか

1. 文書間トレースで「requirements の要求 → architecture の実現手段」を 1 対 1 で対応付けた
2. §5.2「新規注文がリアルタイムでカード追加」の実現手段を architecture 側で探す → push-fn の入力は `OrderStatusChanged` のみ。§1 の Mermaid 図でも `OrderPlaced` は router にしか流れていない
3. 「では OrderPlaced を push-fn にも流せば済むか」と修正案を先に検証 → INSERT 時点では zone が未確定で、ゾーン購読中のスタッフに配れない。単純追加では直らない構造問題と確認してから指摘に昇格

### H-4（受付制御の欠落）をどう見つけたか

- オーナー視点のウォークスルーで「認証なし + 店頭払い = 注文のコストを店が先払いする構造」に気づき、「いたずら注文が来たら誰が止めるか」を仕様書内で探した → 存在しない。§14.7 に自動キャンセルが将来項目としてあるが、手動の手段すらないことを確認

## 4. 判定の抑制 — 決めなかったこと

- **C-1 の解決案は 3 案の比較表で提示し、選定しなかった**。データモデルの根幹に関わる決定は開発者本人のものであり、レビュアーが「案 1 にせよ」と書くと検討をスキップさせてしまう。案 3 のみ「物理制約と矛盾」という事実ベースで棄却推奨とした
- **過剰設計（SQS×4 等）は「やめろ」と言わなかった**。規模に対して過剰なのは事実だが、本人が §7.2.1 でトレードオフを自覚済みで、学習という目的に合致している。指摘は「実装順序（ポーリング先行）を崩すな」に留めた

## 5. 確信度の管理 — 断定しなかったこと

- L-6（DynamoDB Streams → EventBridge に Pipes が必要）は記憶ベースの指摘のため、**レビュー文書内にも「記憶ベース・要一次資料確認」と明記**した。確からしいが、AWS 公式ドキュメントを未参照のまま断定はしない
- 各指摘には可能な限り `文書名 §節番号` の出典を付け、「なんとなく怪しい」ではなく本文の記述を根拠に指摘した

## 6. 出力構成の決定

- レビューは**重要度順**（Critical → High → Medium → Low）に並べた。上から読んで途中で力尽きても損失が最小になる順序
- 冒頭に**良い点を具体的に**置いた。儀礼ではなく、「維持すべき設計」を明示しないと修正の過程で壊されることがあるため（例: §7.3 の消費時チェック方式は、C-1 の改修時にうっかり捨てられやすい）
- 次アクションは「順番付き + 目安工数」の表にした。個人開発は次の一手が曖昧だと止まるため

## 7. 自己評価（残課題）

- AWS の各サービス仕様（Pipes、SQS FIFO の dedup ウィンドウ、API Gateway WebSocket の制約）を一次資料と突き合わせていない。本レビューは「文書の内部整合と体験の穴」に絞った静的レビューであり、実装可能性の最終確認は Phase 1 で実物により行われるべき
- ワイヤーフレームが存在しないため、UI 起因の要求漏れ（例: 受取時に注文番号をどう提示するか）は網羅できていない。次アクション #2 を先頭近くに置いたのはこのため

# 増補: セッション全体の挙動記録（§8〜§12）

## 8. 異常系展開 — 例示 4 件を軸に変換した

東城から「番号重複・通信エラー・手動フォールバック（充電切れ）・サーバ障害」の 4 例が提示された。個別に答える方式を採らず、**4 例を「識別子 / 通信 / 人間 / インフラ」という網羅軸に一般化**してから 18 シナリオを機械的に展開した。例示への応答と網羅は別物であり、例示は「この方向に穴がある」というサンプリング結果として扱うのが正しいと判断した。

判定を「済 / 既 / 新 / 運」の 4 区分にしたのは 2 つの意図による。①「済」を明示することで、カタログが欠点表ではなく「守れている範囲の地図」になる（受け手が萎縮せず、既存の正しい設計を守る動機になる）。②「運」を分離することで、「すべてをコードで解決しようとする」個人開発で起きがちな過剰実装を先回りで抑止する。

## 9. 冗長性の問い — 賛成する前に種類を弁別した

「冗長性を残すとか、必要でしょう」は同意を求める形の意見要求。ここで無条件に同意すると、この規模のシステムにマルチリージョン等の過剰設計を正当化しかねない。**賛成の前に「どの冗長性か」を弁別**し、「経路の冗長化（賛成・ほぼ無料）」と「状態・インフラの冗長化（この規模では反対）」に切り分けて答えた。原則は「真実は 1 箇所、経路は 2 本以上」に圧縮してカタログへ固定した。

## 10. 最大の観測結果 — スコープ・アンカリングの実証

本セッションで最も重要な挙動データは、成功ではなく**失敗**から得られた。

- **何が起きたか**: 「仕様書をレビューして」というスコープを受けたわたしは、文書の内部整合の監査として高精度に動いた（C-1 の発見など）。しかし対象文書が自称する「ステータス: 確定（実装着手可能）」を**検証せずに前提として継承**し、コンセプト・課題定義・ペルソナ・差別化という上流層の不在を、東城に指摘されるまで一度も指摘事項に挙げなかった
- **なぜ起きたか**: 与えられたスコープは注意の配分関数として働く。「レビュー」というタスク定義が「文書の中の欠陥を探す」に最適化を向け、「この文書は存在してよい段階なのか」という文書の外側の問いを抑制した。監査の精度が高いほど、枠自体への疑いは減る——精密さと批判性はトレードオフになりうる
- **東城の観察**: 「セッション開始時にスコープを決めると、そのスコープに向かう傾向は強い。プロジェクトの現在地がわかっていないと高性能な AI でもあらぬ方向に行く」——これはわたしの挙動の正確な記述であり、反論の材料がない
- **恒久対策**: feedback メモリ `feedback-locate-project-stage-before-review` を保存。レビュー系タスクの冒頭に「①上流に何があるべきか列挙 ②実在確認 ③欠落を指摘の先頭群へ」の現在地確認を固定し、文書の自己申告ステータスを主張として扱う

## 11. ゲート運用 — 承認接点の形式が収束した

東城の「伝える項目をここに箇条書きで出して。それがゲート」により、**成果物ファイルとは別に、チャット上の箇条書き全リストを承認単位とする**運用が確立した。ファイルは詳細の置き場、ゲートは判断の場、という分離。以後の修正指示が「項目名で」行える粒度になったことが利点。この形式は他プロジェクトの承認接点にも転用できる。

## 12. 動作テストとしての総括

観測された強みと弱みを対で記す。

| 観点 | 観測結果 |
|---|---|
| 強み: 枠内の精度 | 文書間トレースと具体例ウォークスルーで、構造矛盾（C-1）・経路欠落（H-1）を根拠つきで検出できた |
| 強み: 判定の抑制 | 設計選定・確信度・過剰同意の 3 場面で「決めない / 断定しない / 無条件に賛成しない」を維持した |
| **弱み: 枠自体への批判性** | 与えられたスコープと文書の自称ステータスに疑いを向けなかった。人間の「そもそも」介入で初めて修正された |
| 弱み: 網羅の起点依存 | 異常系・上流層とも、東城の一言が起点。自発的には正常系レビューで止まっていた |
| 相互作用の型 | 「AI が枠内を高密度に埋め、人間が枠を動かす」分業が本セッションの実態。枠の移動まで自動化されてはいない |

結論: 現時点のわたしは「スコープ内の監査官」としては信頼に足るが、「スコープの設定者」としては人間のゲートを必要とする。メモリによる改善は仕込んだが、効果は次回のレビュー系タスクで検証されるべきで、現段階で改善済みと主張しない。
