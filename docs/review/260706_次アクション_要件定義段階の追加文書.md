## 2. 要件定義段階で追加すべき文書（最小構成）

| 文書 | 必須度 | 中身 |
|---|---|---|
| RFP（架空クライアント視点） | **必須** | 背景課題・目的・必須要件・制約（月数百円）・成功基準。全設計判断のトレース先 |
| コンセプト 1 枚 + ペルソナ 4 体 | **必須** | 課題 1 段落・既存アプリとの差別化 1 行・ペルソナ（各ペルソナが検証する仕様と紐づける） |
| 画面一覧・機能一覧 | **必須** | 画面 13 + API 表の読み替え。画面 ⇄ API の突合で欠落（受渡照合・管理系 API）を炙り出す |
| モック 3 系統（顧客スマホ / PC 表示 / 管理画面） | **必須** | 管理画面モックはペルソナ D（非エンジニアのオーナー）の要件を実体化させる装置 |
| 画面遷移図・ワイヤーフレーム | **必須** | UI が決まらないと API 契約の穴（H-2 など）が炙り出せない。レビュー指摘の半分は、画面を描く過程で自然に浮かび上がる類のもの |
| メニューマスタ定義 | **必須** | 商品・バリアント・価格の一覧表（Markdown か spreadsheet） |
| 店舗運用フロー | **必須** | システム外の人間の動きを 1 枚に。支払い突合はここでしか決まらない |
| 受け入れ基準（Phase 別 DoD） | 強く推奨 | 「lint / test 通過」ではなく「実機でこの操作ができる」で書く |
| ADR（`docs/decisions/`） | 推奨 | requirements §14 を 1 判断 1 ファイルに移設。以後は追記専用。§14 をこれ以上肥大化させない |
| 課題管理表 | 推奨 | 軽い Markdown 表でよい。L-4（フェルミ推定 TODO）・L-5（CDK スタック再設計）のような「確定文書に残せない宿題」の受け皿 |

**作らなくてよいもの**: 恒久的な詳細設計書（コードと二重管理になり腐る。architecture.md の「実装契約」の粒度で止めるのが正しい）、WBS、運用手順書（MVP 後でよい）、テスト仕様書の網羅版（DoD で代替）。

## 3. 進め方のアドバイス

- **ポーリング版（Phase 1〜6）を先に一周する**。WebSocket / SQS / EventBridge（Phase A〜C)は「動くもの」ができてから積む。requirements §14.8 の順序は正しいので信じてよい
- 各 Phase 末に **実機（スマホ実機 + タブレット）で触って確認**してから次へ。lint / test / build 通過は必要条件であって完了条件ではない
- 設計判断が発生するたびに ADR を 1 枚追加する。§14 を書いた習慣をそのまま継続する
- 詰まったら「この機能は誰がどう使うか」に立ち返る。今回の Critical（C-1）も「ラテとお茶を一緒に頼んだ客」を想像すれば見つかる種類の問題だった
