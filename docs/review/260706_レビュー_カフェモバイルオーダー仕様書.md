## 総評

個人開発の設計として水準は高い。特に以下は先回りが効いており、そのまま維持してよい。

- `Idempotency-Key`・サーバ側価格再計算・条件付き書き込みによる遷移検証 — 事故が起きやすい箇所を最初から塞いでいる
- キャンセル競合の「消費時チェック」方式（requirements.md §7.3）— SQS メッセージ削除に走らなかった判断は正しい
- §14 の設計判断記録 — 確定仕様の「なぜ」が残っており、将来の再検討コストを大きく下げる
- アダプタ層 / ドメイン層の分離（architecture.md §4）と共通エラーエンベロープ（§5.1）
- Phase 1〜6 を「ポーリングで一周してからリアルタイム化」する実装順序（requirements.md §14.8）

ただし、**「注文単位のステータス」と「明細単位のゾーン振り分け」が構造的に矛盾しており（C-1）**、これを未解決のまま Phase 4（注文API）に入ると手戻りが大きい。着工前に決めるべき事項が 3 件ある（C-1 / H-2 / H-3）。

## 指摘一覧（重要度順）

### [Critical] C-1: 注文単位ステータス × 明細単位ルーティングの矛盾

- **事実**: `machine-router-fn` は「**明細ごとに**ゾーン判定」する（architecture.md §8.2、`MessageDeduplicationId = orderId+lineIndex`）。一方、ステータス（§6）・QueuePosition の `zone`（architecture.md §6.4）・スタッフのカード操作（§5.3）はすべて**注文単位**。
- **失敗シナリオ**: 「カフェラテ L 1杯 + 紅茶 M 1杯」の注文 → Zone A と Zone D に 1 明細ずつ入る。
  1. `QueuePosition` は `"zone": "B"` のように 1 ゾーンしか返せない
  2. Zone A のバリスタが完成スワイプすると注文全体が `READY_PICKUP` になり、紅茶が未着手でも顧客に「受取待ち」が表示される
  3. スタッフカードがどちらのゾーンに表示されるか未定義
- **選択肢**（選定は設計者自身で行う。データモデルに効くため Phase 1 着工前に）:

| 案 | 内容 | トレードオフ |
|---|---|---|
| 案1 | 製造単位を「明細（ライン）」に変更。ラインごとに status / zone / queueSeq を持ち、注文 status は導出（全ライン READY で `READY_PICKUP`）。スタッフカード = ライン | 実運用に最も近い。データモデル・API・UI に中規模の改修 |
| 案2 | MVP 制約として「1 注文 = 1 明細」に割り切る（複数杯は複数注文） | モデル無傷で最速。体験は落ちるが個人開発 MVP としては合理的 |
| 案3 | 注文単位で代表ゾーンに一括ルーティング | L 専用マシンで M は作れない＝ゾーン定義（§7.1）と物理的に矛盾。**棄却推奨** |

### [High] H-1: スタッフへの「新規注文」push の経路が未定義

- requirements.md §5.2 は「新規注文が入るとリアルタイム（WebSocket）でカードが追加される」を要求。しかし `push-fn` の入力は `OrderStatusChanged` のみ（architecture.md §8.3、§1 図では `EB -->|OrderStatusChanged| PUSH`）。`OrderPlaced` → push の経路が存在しない。
- さらに構造的な問題: INSERT 時点では `zone` 未確定（`zone-consumer-fn` が後から書く）ため、ゾーン購読中のスタッフ（`connect-fn ?zone=`）宛に配りようがない。
- **修正方向**: zone 確定の書き込み（Streams MODIFY）を契機に `OrderZoneAssigned` 相当のイベントを定義して push する。C-1 の決定と同時に設計する。

### [High] H-2: キュー位置の算出方法とアクセスパターンの欠落

- `GET /orders/{orderId}/queue-position` は `position` を返すが、**算出方法がどちらの文書にも定義されていない**。「ゾーン内で自分より前の未完了注文数」を数えるには「ゾーン別 × queueSeq 順」の照会が必要だが、AP1〜AP8（requirements.md §11.3）に該当パターンがなく、GSI1 のキーは `STORE#<storeId>#<status>` × `createdAt`。
- 併発する問題 2 点:
  - スタッフ一覧 API（`GET /stores/{storeId}/orders`）に `?zone=` パラメータがなく、§5.1「選択ゾーンの注文のみ表示」を満たせない
  - 注文直後〜zone 確定までの非同期の窓で queue-position が何を返すか未定義（404？ pending？）
- **修正方向**: GSI2（`ZONE#<zone>` × `queueSeq`）を 1 本追加するか GSI1 キーを再設計。「zone 未確定時は `"position": null, "state": "assigning"` を返す」等を契約に明記。

### [High] H-3: スタッフ側のキャンセル・巻き戻し手段がない

- `store-fn` は前進遷移のみ（architecture.md §6.5）。顧客キャンセルは `STORE_ACCEPTED` のみ（§4.5）。
- **失敗シナリオ**: 作り間違い・受取放置・いたずら注文をスタッフが処分できない。アンドゥバナーの 3 秒を過ぎた誤操作も戻せない。詰んだ注文がキューに残り続ける。
- **修正方向**: スタッフ用の `CANCELLED` 遷移（`PREPARING` / `READY_PICKUP` からも可）を要件に追加。放置注文の自動キャンセル（§14.7）は将来でよいが、**手動の手段は MVP 必須**。

### [High] H-4: 受付制御（営業時間・受付停止）がない

- §2.1 に営業時間の定義はあるが、**時間外の注文をサーバ側で拒否する要件がない**。ピークで捌けない時に店側が「受付一時停止」する手段もない。
- 認証なし＋店頭払いのため、いたずら・No-show 注文のコスト（作ったドリンクの廃棄）は店が全額負う。オーナー視点で最も痛い欠落。
- **修正方向**: 店舗レコードに受付停止フラグ + 営業時間チェックを MVP 要件化。同一 `sessionId` の未完了注文数上限も 1 行入れる。

### [Medium] M-1: ConnectionTable の逆引きができない

- PK=`connectionId`, SK=`orderId`（architecture.md §7）。`push-fn` は **orderId → connectionId** を引く必要があるが、このキー設計では Query できず Scan になる。GSI 追加かキー逆転（PK=`orderId`）が必要。

### [Medium] M-2: EMA 更新箇所の文書間矛盾

- requirements.md §8.1 の計測点は `PREPARING → READY_PICKUP` 遷移（= store-fn 経由のステータス変化時にしか所要時間は確定しない）。だが architecture.md §8.4 では**注文投入時に走る** `zone-consumer-fn` が「ETA 算出用の基準（ゾーン別 EMA の入力）を更新」と記載。投入時点で製造所要時間は知り得ない。EMA 更新は `OrderStatusChanged` 側の消費者に置くのが整合的。

### [Medium] M-3: メニュー管理手段が未定義

- `Product.available` フィールドはあるが、売り切れ・メニュー変更を**誰がどう反映するか**（管理 API / DynamoDB 直接編集 / スクリプト）が未定義。MVP は「seed スクリプト + 手動更新」でよいが、それを明記する。

### [Medium] M-4: スタッフ表示順が SQS FIFO 投資と不整合

- §7.2.1 で「`createdAt` 順は同着・時計ずれで信用できないから FIFO を採用」と論証しながら、§5.2 のスタッフ表示順は「注文受付時刻の昇順」。表示も `queueSeq` 順に統一しないと、FIFO で守った公平性が画面で崩れる。

### [Low] 細目

| # | 内容 |
|---|---|
| L-1 | 注文番号（#042）の採番方式が未定義（日次リセット？店舗単位のアトミックカウンタ？） |
| L-2 | カート・Idempotency レコードの TTL 未定義（無限に溜まる。ConnectionTable と同様 TTL を） |
| L-3 | `PATCH /orders/{id}/cancel` は動詞サブリソースなので `POST` が一般的（動作影響なし） |
| L-4 | §2.1 に「フェルミ推定的に算出したい」の TODO が残ったままステータスが「確定」 |
| L-5 | §14.9 の CDK スタック再設計が未消化（課題管理表の受け皿へ） |
| L-6 | DynamoDB Streams → EventBridge は直結できず、**EventBridge Pipes** を挟むか **Streams → Lambda 直接トリガー**にする必要があるはず。この指摘自体も記憶ベースなので、実装前に AWS 公式ドキュメントで要確認 |

## 規模感についての所見

1 日 500 人・同時 100 接続に対して SQS FIFO ×4 + EventBridge + Streams は明らかに過剰装備だが、§7.2.1 でトレードオフを自覚して選んでおり、学習目的の個人開発としては妥当な選択。**Phase 1〜6（ポーリング版）を先に一周して動くものを作る計画（§14.8）は正しいので、その順序を崩さないこと。**
