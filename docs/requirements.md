## 1. プロジェクト概要

### 1.1 背景とゴール

カフェ利用者が来店前・来店と同時にスマートフォンで注文を完了し、到着と同時に商品を受け取れる体験を実現する。

- **形態**: テイクアウト専用の、アプリのように使えるWebサイト（＝PWA）
- **物理構成**: バリスタが操作する 4 台の製造マシン（ゾーン A/B/C/D）に注文を自動振り分け
- **価値提案**: 注文待ち行列の可視化。そして単一のレーン作業ではなく、ドリンクごとに最適化されたレーンへの振り分けにより、従来のカフェよりも「並ばないカフェ体験」を提供

### 1.2 開発方針

- **グリーンフィールド（１から作る新規開発）**: 既存資産への依存なし。新規リポジトリ `mobile-order-app` で一から構築
- **アーキテクチャ方針**: AWS サーバーレス（Lambda / DynamoDB / API Gateway）で構築。Well-Architected に準拠した設計。
- **コスト方針**: アイドル時ほぼゼロのサーバーレス従量課金。月数百円以内を目標
- **技術スタック**: フロントエンド（Next.js + PWA）/ バックエンド（Go Lambda）/ インフラ（AWS CDK TypeScript）

## 4. 機能要件 — 顧客向け

### 4.1 メニュー閲覧

- カテゴリ別にメニューを一覧表示する
- 各商品に以下の組み合わせ（バリアント）を設定できる


| バリアント | 選択肢                          |
| ----- | ---------------------------- |
| 個数    | 1〜上限（サーバー側で検証）               |
| 温度    | ホット / アイス（商品によって制限あり）        |
| サイズ   | S / M / L（エスプレッソ非使用はS / Mのみ） |


### 4.2 カート

- 商品をカートに追加できる
- カート内の個数を変更できる
- カート内の明細を削除できる
- 同一商品でも温度・サイズの組み合わせが異なれば別明細として扱う

### 4.3 注文

- 「カート追加」または「すぐ注文」の 2 方式で注文を開始できる
- 注文確定時に以下を発行する
  - 注文番号（顧客に表示する短い連番）
  - 注文 ID（システム内部で使う、重複しない管理番号（UUID））
- 注文確定後、金額はサーバー側で再計算する（クライアント送信値を信用しない）
- 二重注文防止のため、二重送信防止用の合言葉（`Idempotency-Key`）に対応する

### 4.4 注文状態の確認

- 注文完了後、リアルタイムで製造ステータスを確認できる
  - フォアグラウンド時: WebSocket push で状態変化を即時受信
  - バックグラウンド復帰時: REST API で最新状態を再取得 + WebSocket 再接続
- 自分の前に何件の注文があるかを表示できる（キュー位置）
  - フォアグラウンド中に 30 秒間隔で定期的な問い合わせ（ポーリング）により更新

### 4.5 注文キャンセル

- `STORE_ACCEPTED` 状態の注文に限り、顧客自身がキャンセルできる
- `PREPARING` 以降はキャンセル不可
- キャンセル後、該当注文はマシンの製造キューから除外される

## 5. 機能要件 — スタッフ向け

### 5.1 ログイン・ゾーン選択

```
/staff (Cognito ログイン画面)
  ↓ 認証成功
ゾーン選択画面（A / B / C / D）
  ↓ ゾーン選択
注文キュー画面（選択ゾーンの注文のみ表示）
```

### 5.2 注文キュー表示

- 担当ゾーンに割り当てられた注文をカード形式で一覧表示する
- 新規注文が入るとリアルタイム（WebSocket）でカードが追加される
- 表示順: 注文受付時刻の昇順（古い順）

### 5.3 スワイプ操作（Framer Motion 実装）


| 操作        | アクション                      |
| --------- | -------------------------- |
| 右スワイプ     | 製造開始（STORE_ACCEPTED → PREPARING） |
| （完成時）スワイプ | 製造完了（PREPARING → READY_PICKUP）    |
| 受取確認スワイプ  | 受渡完了（READY_PICKUP → HANDED_OVER）    |


### 5.4 誤操作対策（B案：アンドゥバナー）

1. スワイプ操作後、画面下部に「取り消し」バナーを 3 秒間表示
2. 3 秒以内に「取り消し」をタップ → 状態変化をキャンセル（DynamoDB への書き込みは行わない）
3. 3 秒経過後 → DynamoDB に書き込み確定、WebSocket で顧客へ通知

## 6. 注文ステータス仕様

### 6.1 ステータス遷移図

```
（将来）PENDING_PAYMENT ──→ STORE_ACCEPTED
                             │
                             ↓
                          PREPARING
                             │
                             ↓
                           READY_PICKUP
                             │
                             ↓
                         HANDED_OVER

STORE_ACCEPTED ──→ CANCELLED（顧客キャンセル。STORE_ACCEPTED のみ可）
```

### 6.2 ステータス定義


| ステータス             | 説明           | 遷移元             | 遷移先                   |
| ----------------- | ------------ | --------------- | --------------------- |
| `STORE_ACCEPTED`        | 注文受付済み（製造待ち） | 注文確定            | PREPARING / CANCELLED |
| `PREPARING`       | 製造中          | STORE_ACCEPTED        | READY_PICKUP                 |
| `READY_PICKUP`           | 製造完了（受取待ち）   | PREPARING       | HANDED_OVER             |
| `HANDED_OVER`       | 受渡済み         | READY_PICKUP           | —                     |
| `CANCELLED`       | キャンセル済み      | STORE_ACCEPTED        | —                     |
| `PENDING_PAYMENT` | 決済待ち（予約定義のみ） | —               | STORE_ACCEPTED              |
| `PAYMENT_FAILED`  | 決済失敗（予約定義のみ） | PENDING_PAYMENT | —                     |

## 7. マシンゾーン仕様

### 7.1 ゾーン定義


| ゾーン        | 対象商品                     | サイズ   | 物理マシン台数 |
| ---------- | ------------------------ | ----- | ------- |
| **Zone A** | エスプレッソ系                  | L のみ  | 1 台     |
| **Zone B** | エスプレッソ系                  | M のみ  | 1 台     |
| **Zone C** | エスプレッソ系                  | S のみ  | 1 台     |
| **Zone D** | エスプレッソ不使用（ティー系・フラペチーノ系等） | S / M | 1 台     |


### 7.2 振り分けロジック（MVP: ルールベース）

注文確定（DynamoDB への書き込み）を **DynamoDB Streams(INSERT) → EventBridge** が捕捉して `machine-router-fn` を起動する。同関数がドリンクカテゴリ + サイズから自動でゾーンを決定し、対応する **SQS FIFO キュー**（ゾーンごとに 1 本 = 計 4 本）へエンキューする。実際にキューを消費（consume）するのは `zone-consumer-fn`（プログラム）であり、バリスタ（人間）ではない。

```
エスプレッソ系 × L → Zone A
エスプレッソ系 × M → Zone B
エスプレッソ系 × S → Zone C
エスプレッソ不使用 × S/M → Zone D
```

### 7.2.1 なぜ SQS FIFO を採用するか（案A採用の根拠）

ゾーンへの振り分けは「DynamoDB に受付順（`createdAt`）を持たせるだけ（案B）」でもこの規模では実用上成立する。しかし本設計では以下の理由から **SQS FIFO を採用（案A）** する。


| 観点             | SQS FIFO 採用の効果                                                                         |
| -------------- | -------------------------------------------------------------------------------------- |
| **公平性（受付順保証）** | 同一ゾーン内で「先に注文した客が先に受け取れる」順序を AWS 基盤レベルで厳密に保証する。案B（`createdAt` 順）は同一ミリ秒の同着や時計ずれで順序が逆転しうる |
| **疎結合・バッファ**   | 注文書き込み（order-fn）と製造順の確定を分離できる。昼ピークの注文集中をキューで吸収する                                       |
| **重複排除**       | FIFO の重複排除（dedup）により二重エンキューを防ぐ（`Idempotency-Key` と二重の防御）                               |
| **将来拡張**       | 自動製造マシンを導入する際、キューをそのまま製造制御の入力として流用できる                                                  |


**トレードオフと結論**: `zone-consumer-fn` と 4 本のキューという部品増を伴う。案B なら部品ゼロで済むが、**受付順の厳密な公平性**と**将来の自動製造連携**を優先し、案A を採用する。

**消費モデルの注意**: SQS を取り出す（consume）のは `zone-consumer-fn`（プログラム）であり、バリスタ（人間）ではない。バリスタは DynamoDB を裏に持つスタッフ UI を見て操作する（§5）。SQS は「受付順の確定」に用い、一覧表示・検索は DynamoDB（GSI1）が担う、という役割分担とする。

### 7.3 キャンセル時のキュー除外

- キャンセルリクエスト受信時、DynamoDB の `status` を `CANCELLED` に更新する（条件: `status = STORE_ACCEPTED`）
- SQS からのメッセージ削除は行わない
- `zone-consumer-fn`（SQS コンシューマ）は処理前に DynamoDB の `status` を確認し、`CANCELLED` の場合はスキップする

## 8. ETA（製造完了予測）仕様

### 8.1 計測ポイント


| イベント       | タイミング                              |
| ---------- | ---------------------------------- |
| 製造開始時刻     | スタッフが「製造開始」スワイプ → `PREPARING` 遷移時刻 |
| 製造完了時刻     | スタッフが「完成」スワイプ → `READY_PICKUP` 遷移時刻       |
| エスプレッソ抽出時間 | 固定 25 秒（定数として扱い、学習対象から除外）          |


### 8.2 フェーズ 1（MVP）: ゾーン単位の指数移動平均

- 製造時間の実績をゾーン単位で蓄積
- 指数移動平均（EMA）で予測値を更新
- `GET /orders/{orderId}/queue-position` のレスポンスに `estimatedWaitMinutes` として返す

### 8.3 フェーズ 2（将来・データ 300〜500 件蓄積後）: Amazon Forecast

- ゾーン単位の細分化 → 商品単位の予測へ移行
- Amazon Forecast による時系列予測モデルの導入

## 9. リアルタイム通知仕様

### 9.1 WebSocket の動作


| 状態               | 動作                                                                                     |
| ---------------- | -------------------------------------------------------------------------------------- |
| フォアグラウンド（アプリ表示中） | WebSocket 接続を維持。ステータス変化を即時 push                                                        |
| バックグラウンド移行時      | OS がタブを停止 → WebSocket 自動切断（アプリは何もしない）                                                  |
| バックグラウンドからの復帰    | `Page Visibility API`（`visibilitychange` イベント）で検知 → REST API で最新状態を再取得 → WebSocket 再接続 |


### 9.2 WebSocket の活用範囲


| 利用者  | 通知内容                                       |
| ---- | ------------------------------------------ |
| 顧客   | 注文ステータスの変化（STORE_ACCEPTED → PREPARING → READY_PICKUP 等） |
| スタッフ | 新規注文の到着（新しいカードのリアルタイム追加）                   |


### 9.3 キュー位置のポーリング

- WebSocket では push しない（位置は注文状態ではなくキューの動的な状態）
- フォアグラウンド中のみ 30 秒間隔でポーリングを行う
- バックグラウンド時はポーリングを停止

## 10. 非機能要件

### 10.1 可用性

- AWS マネージドサービス（Lambda / DynamoDB / API Gateway）の AZ 冗長を活用
- SLA 目標: 99.9%（単一リージョン・単一 AZ 障害で継続稼働）

### 10.2 パフォーマンス

- API レスポンスタイム（P95）: 500ms 以内（Lambda コールドスタートを除く）
- WebSocket 通知遅延: 1 秒以内（DynamoDB Streams → EventBridge → push-fn → クライアント）

### 10.3 スケーラビリティ

- Lambda + DynamoDB オンデマンドにより、100 同時接続・昼ピーク時も自動スケール

### 10.4 セキュリティ


| 要件           | 対応                                        |
| ------------ | ----------------------------------------- |
| 通信経路の暗号化     | CloudFront / API Gateway で HTTPS 強制       |
| 保存データの暗号化    | DynamoDB デフォルト暗号化 / S3 SSE-S3             |
| 最小権限 IAM     | Lambda ごとに必要な DynamoDB 操作のみ許可（CDK grants） |
| 入力バリデーション    | サーバー側で enum 検証・個数上限・価格再計算                 |
| 二重注文防止       | `POST /orders` は `Idempotency-Key` に対応    |
| スロットリング      | API Gateway でバースト / レート上限を設定              |
| Staff API 認証 | Cognito JWT オーソライザー必須                     |


### 10.5 コスト

- アイドル時（夜間・週末）ほぼ ¥0
- 通常稼働時の目標: 月額 ¥1,000 未満


| サービス                 | 月額概算                     |
| -------------------- | ------------------------ |
| Lambda               | ¥0（無料枠 100 万 req/月 内）    |
| API Gateway HTTP API | ¥0〜数百円（$1.0 / 100 万 req） |
| DynamoDB オンデマンド      | 数百円                      |
| S3 + CloudFront      | 数十〜数百円                   |
| CloudWatch           | 数十円                      |
| **合計**               | **¥0〜数百円（$5 未満）**        |


### 10.6 可観測性

- 全 Lambda から CloudWatch Logs へ構造化 JSON ログを出力
- CloudWatch メトリクス / アラーム: 4xx / 5xx 率、Lambda エラー率、DynamoDB スロットル
- WebSocket 接続数・SQS キュー深度を監視

## 11. API 設計

### 11.1 エンドポイント一覧

#### Customer API（認証なし）


| メソッド   | パス                                 | Lambda    | 説明                       |
| ------ | ---------------------------------- | --------- | ------------------------ |
| GET    | `/menu`                            | menu-fn   | カテゴリ別メニュー一覧              |
| GET    | `/menu/{productId}`                | menu-fn   | 商品詳細                     |
| GET    | `/cart/{sessionId}`                | cart-fn   | カート取得                    |
| POST   | `/cart/{sessionId}/items`          | cart-fn   | カートに商品を追加                |
| PUT    | `/cart/{sessionId}/items/{itemId}` | cart-fn   | カート明細の個数変更               |
| DELETE | `/cart/{sessionId}/items/{itemId}` | cart-fn   | カート明細の削除                 |
| POST   | `/orders`                          | order-fn  | 注文確定（Idempotency-Key 対応） |
| GET    | `/orders/{orderId}`                | status-fn | 注文ステータスのポーリング            |
| GET    | `/orders/{orderId}/queue-position` | status-fn | キュー位置と待ち時間の取得            |
| PATCH  | `/orders/{orderId}/cancel`         | order-fn  | 注文キャンセル（STORE_ACCEPTED のみ）     |


#### Staff API（Cognito 認証必須）


| メソッド  | パス                         | Lambda   | 説明                                     |
| ----- | -------------------------- | -------- | -------------------------------------- |
| GET   | `/stores/{storeId}/orders` | store-fn | ステータス別注文一覧（`?status=` クエリ）             |
| PATCH | `/orders/{orderId}/status` | store-fn | ステータス更新（PREPARING / READY_PICKUP / HANDED_OVER） |


#### WebSocket API


| ルート           | Lambda        | 説明                                               |
| ------------- | ------------- | ------------------------------------------------ |
| `$connect`    | connect-fn    | connectionId を ConnectionTable に保存               |
| `$disconnect` | disconnect-fn | ConnectionTable から connectionId を削除              |
| `$default`    | push-fn       | クライアント → サーバー送信の受け口（当面スタブ。push は EventBridge 駆動） |


> **補足（インターフェース上の注記）**
>
> - `GET /menu/{productId}`: menu-fn は全メニュー（小規模・キャッシュ可能）を取得し、その中から該当商品を返す。パスに category を含めなくてよい。
> - **push-fn**: サーバー → クライアントの push は **EventBridge 駆動**（DynamoDB Streams のステータス変化 → EventBridge → PostToConnection）で行う。WebSocket の `$default` は当面クライアント → サーバー送信のスタブに留める。
> - **machine-router-fn / zone-consumer-fn**: 注文の書き込みを **DynamoDB Streams → EventBridge** で捕捉し、machine-router-fn がゾーン判定 → SQS FIFO へエンキュー。`zone-consumer-fn` が各キューを消費して受付順を確定する。

### 11.2 Lambda 関数一覧


| 関数名                 | 属する API         | 役割                                                                                 |
| ------------------- | --------------- | ---------------------------------------------------------------------------------- |
| `menu-fn`           | Customer        | メニュー取得                                                                             |
| `cart-fn`           | Customer        | カート操作                                                                              |
| `order-fn`          | Customer        | 注文確定・キャンセル                                                                         |
| `status-fn`         | Customer        | ステータスポーリング・キュー位置                                                                   |
| `store-fn`          | Staff           | 注文一覧・ステータス更新                                                                       |
| `machine-router-fn` | 内部（EventBridge） | ゾーン A〜D への SQS 振り分け                                                                |
| `zone-consumer-fn`  | 内部（SQS FIFO）    | 4 ゾーンの SQS FIFO を単一 Lambda で消費し、受付順（キュー位置）と ETA 基準を DynamoDB に確定。`CANCELLED` はスキップ |
| `connect-fn`        | WebSocket       | 接続時の connectionId 保存                                                               |

### 12.1 アプリ内決済 導入時のフロー（将来実装時の詳細）

MVP では「商品選択（メニュー・カート）→ 注文 → `STORE_ACCEPTED`（注文受付＝製造キューへの振り分け対象）」という、決済ステップを持たないフローで実装する。商用化にあたってアプリ内決済を導入する際は、`STORE_ACCEPTED` の手前に決済ステップを挟み込む形で、以下の手順を想定する。

1. `POST /orders` → 注文を `PENDING_PAYMENT`（決済待ち）状態で作成し、`orderId` を返す
2. クライアントを決済プロバイダ（Stripe 等）のチェックアウト画面へリダイレクトする
3. 決済完了 → Stripe Webhook を受信 → `payment-fn` が注文を `STORE_ACCEPTED` に更新し、WebSocket で push する
4. クライアントがチェックアウトから復帰 → WebSocket で状態変化を受信 → キュー位置の表示を開始する

- 決済失敗時は `PAYMENT_FAILED` に遷移し、カート内容は保持する（再決済できるようにするため）
- 「商品選択」自体は決済の有無に関わらず共通のフローであり、決済ステップだけを注文確定と製造キュー投入の間に後付けで挟み込める設計とする（§14.1 の Cognito 格上げと同様、`order-fn` 自体の構造は変えずに済む）

## 14. 主要な設計判断の経緯

本セクションは確定仕様の「なぜ」を記録し、将来の再検討時に判断基準を参照できるようにするためのもの。旧プランファイル（`nifty-wishing-origami.md` / `todo-snug-scott.md`）に記録されていた議論の要点をここに統合する。

### 14.1 Cognito 認証の位置づけ変更

- 初期設計ではスコープ外（将来の「特別メニュー」機能向けの布石）だった
- スタッフ用画面（ゾーン選択・注文管理）へのアクセス制御が別途必要と判明し、**MVP必須に格上げ**した
- 個人別アカウントではなく共有アカウント1つ（最大4台タブレット同時ログイン）とし、運用をシンプルに保った

### 14.2 リアルタイム通知に WebSocket を採用（ポーリングではなく）

- Well-Architected の「パフォーマンス効率」の観点。ポーリングよりリクエスト数が減り、状態変化時のみ通信するため無駄な DynamoDB Read が発生しない
- バックグラウンドでの WebSocket 維持はブラウザ仕様上不可能なため、`Page Visibility API` による復帰時の再接続方式を採用
- Web Push（FCM経由）は通知許可のUXコストがあるため、まずはフォアグラウンド前提の設計とし将来拡張に回した

### 14.3 注文キャンセル方式の比較と選定


| 方式                                   | 速度  | コスト          | 複雑さ | 判定                                 |
| ------------------------------------ | --- | ------------ | --- | ---------------------------------- |
| A. DynamoDB 条件付き Update + 消費側の冪等チェック | 高   | 低（追加コストほぼゼロ） | 低   | **採用**                             |
| B. SQS メッセージ削除                       | 中   | 中            | 中   | 見送り（`ReceiptHandle` の保存コストが発生するため） |
| C. DynamoDB Streams でキャンセルイベントを伝播    | 低   | 低            | 高   | 見送り（複雑さが見合わない）                     |


### 14.4 マシンルーティング: ルールベース先行、Bedrock は段階導入

- MVP は if / switch の**ルールベース**で十分と判断（実装・テストが容易）
- Bedrock は「キュー深度に応じた動的振り分け」「処理時間の異常検知」など複雑な最適化が必要になった段階（フェーズ2）で初めて価値が出るため、データが溜まってから導入判断する

### 14.5 ETA 予測: EMA 先行、Amazon Forecast は段階導入（Bedrock は不採用）

- ETA 予測は時系列予測タスクであるため、汎用 LLM の Bedrock ではなく専用の時系列予測サービスである **Amazon Forecast** を採用先として選定。
- データが 300〜500 件蓄積されるまでは、フェーズ1としてゾーン単位（A, B, C, Dの４つ）の指数移動平均（EMA）で代替する段階的導入方針とした。

### 14.6 EventBridge と SQS の使い分け方針

- **SQS FIFO**: 「送り主1つ→受け取り先1つへ、順番を守って渡したいだけ」の処理に使う。ゾーンごとの製造キュー（1台のマシンへ注文を受付順に渡す）がこれに当たる
- **EventBridge**: 「1つの出来事（例: 注文が入った、ステータスが変わった）を、複数の異なる担当処理に同時に知らせたい」場合に使う。「注文が入った」という1つの出来事をきっかけに、マシンへの振り分け・顧客への通知・（将来の）売上集計・（将来の）ポイント付与など、複数の処理を同時に走らせられる
- この使い分けにより、将来新しい機能（通知・ポイント付与・売上集計・外部POS連携など）を追加したくなっても、注文を受け付ける `order-fn` 自体のコードは変更せず、EventBridge側に新しい「知らせる相手」を追加登録するだけで対応できる（§7.2, §7.2.1 参照）
