## 1. プロジェクト概要

### 1.1 背景とゴール

カフェ利用者が来店前・来店と同時にスマートフォンで注文を完了し、到着と同時に商品を受け取れる体験を実現する。

- **形態**: テイクアウト専用の、アプリのように使えるWebサイト（＝PWA）
- **物理構成**: バリスタが操作する 4 台の製造マシン（ゾーン A/B/C/D）に注文を自動振り分け。完成品は自動搬送レーンで提供台へ運ばれ、既存の自動受渡システム（電子パネル＋検知カメラ）がお客様への受け渡しを検知する（§7.4）
- **価値提案**: 注文待ち行列の可視化。そして単一のレーン作業ではなく、ドリンクごとに最適化されたレーンへの振り分けにより、従来のカフェよりも「並ばないカフェ体験」を提供

### 1.2 開発方針

- **グリーンフィールド（１から作る新規開発）**: 既存資産への依存なし。新規リポジトリ `mobile-order-app` で一から構築
- **アーキテクチャ方針**: AWS サーバーレス（Lambda / DynamoDB / API Gateway）で構築。Well-Architected に準拠した設計。
- **コスト方針**: アイドル時ほぼゼロのサーバーレス従量課金。月数百円以内を目標
- **技術スタック**: フロントエンド（Next.js + PWA）/ バックエンド（Python Lambda）/ インフラ（AWS CDK TypeScript）

## 3. ユーザー種別とアクセス方式

### 3.1 顧客


| 項目     | 仕様                              |
| ------ | ------------------------------- |
| アクセス方法 | 店内 QR コード読み取りなど、既知 URL から直接アクセス |
| 認証     | 不要（MVP では sessionId で識別）        |
| デバイス   | スマートフォン（ブラウザ / PWA）             |
| URL    | `cafe.example.com/`             |


### 3.2 スタッフ

スタッフが担当する役割は、ゾーン担当（Zone A〜D、製造）の4択。従来検討していた提供台担当というログインロールは廃止し、受け渡し工程は既存の自動受渡システム（電子パネル＋検知カメラ、§7.4）に置き換える。


| 項目       | 仕様                                            |
| -------- | --------------------------------------------- |
| アクセス方法   | `cafe.example.com/staff` にアクセス → Cognito ログイン |
| 認証       | Cognito（共有アカウント 1 つ。オーナーがコンソール画面から手動作成）       |
| 同時ログイン   | 最大 4 台のタブレットが同じアカウントで同時ログイン可。各タブレットはログインのたびに A/B/C/D のいずれかを選ぶ（§5.1） |
| デバイス     | タブレット（スマホならなんでも可能）                            |
| 個人別アカウント | スコープ外（将来の拡張）                                  |

## 4. 機能要件 — 顧客向け

### 4.1 メニュー閲覧

- カテゴリ別にメニューを一覧表示する
- 各商品に以下の組み合わせ（バリアント）を設定できる


| バリアント | 選択肢                          |
| ----- | ---------------------------- |
| 個数    | 1〜上限（サーバー側で検証）               |
| 温度    | ホット / アイス（商品によって制限あり）        |
| サイズ   | S / M / L（エスプレッソ非使用はS / Mのみ） |


### 4.2 カート

- 商品をカートに追加できる
- カート内の個数を変更できる
- カート内の明細を削除できる
- 同一商品でも温度・サイズの組み合わせが異なれば別明細として扱う

### 4.3 注文

- 「カート追加」または「すぐ注文」の 2 方式で注文を開始できる
- 注文確定時に以下を発行する
  - 注文番号（顧客に表示する短い連番）
  - 注文 ID（システム内部で使う、重複しない管理番号（UUID））
- 注文確定後、金額はサーバー側で再計算する（クライアント送信値を信用しない）
- 二重注文防止のため、二重送信防止用の合言葉（`Idempotency-Key`）に対応する

### 4.4 注文状態の確認

- 注文完了後、リアルタイムで製造ステータスを確認できる
  - フォアグラウンド時: WebSocket push で状態変化を即時受信
  - バックグラウンド復帰時: REST API で最新状態を再取得 + WebSocket 再接続
- 自分の前に何件の注文があるかを表示できる（キュー位置）
  - フォアグラウンド中に 30 秒間隔で定期的な問い合わせ（ポーリング）により更新

### 4.5 注文キャンセル

- 注文に含まれる**全明細が `WAITING`**（§6.1）の間に限り、顧客自身が注文全体をキャンセルできる
- 1 明細でも `PREPARING` 以降に進んでいる場合、注文全体のキャンセルは不可（一部のゾーンで製造着手済みのため）
- キャンセル後、該当注文の全明細はマシンの製造キューから除外される

## 5. 機能要件 — スタッフ向け

### 5.1 ログイン・選択

タブレットは特定のゾーンに固定配置されるとは限らない。そのため、**ログインのたびに毎回**、A / B / C / D の中から今回自分が担当する持ち場を選ぶ（固定の役割や記憶された選択はない）。提供台での受け渡しはスタッフのログイン操作を伴わない（§7.4）。

```
/staff (Cognito ログイン画面)
  ↓ 認証成功
選択画面（A ｜ B ｜ C ｜ D）← 単一のフラットな一覧。毎回選び直す
  └─ A/B/C/D を選択 → 注文キュー画面（選択ゾーンの明細のみ表示、§5.2）
```

### 5.2 注文キュー表示（ゾーン担当）

- 担当ゾーンに割り当てられた明細をカード形式で一覧表示する（AP-Z1、`architecture.md` §6.3）
- 新規明細が入るとリアルタイム（WebSocket）でカードが追加される
- 表示順: ゾーン内の受付順（`queueSeq` 昇順）

### 5.3 スワイプ操作（Framer Motion 実装、ゾーン担当）

「製造開始」「完成」の 2 操作は、ゾーン担当スタッフが**自分のゾーンに割り当てられた明細（商品 1 品目）単位**で行う（§6.1）。担当ゾーンの画面には、自分のゾーンの明細だけが表示される。


| 操作        | アクション                      | 単位 |
| --------- | -------------------------- | --- |
| 右スワイプ     | 製造開始（WAITING → PREPARING） | 明細（商品1品目） |
| （完成時）スワイプ | 製造完了（PREPARING → READY）    | 明細（商品1品目） |

完成した明細は自動搬送レーンに乗せ、提供台へ送る（§7.4）。ゾーン担当はこの時点で当該明細の操作を終える。

### 5.4 誤操作対策（B案：アンドゥバナー、ゾーン担当）

1. スワイプ操作後、画面下部に「取り消し」バナーを 3 秒間表示
2. 3 秒以内に「取り消し」をタップ → 状態変化をキャンセル（DynamoDB への書き込みは行わない）
3. 3 秒経過後 → DynamoDB に書き込み確定、WebSocket で顧客へ通知

### 5.5 受渡確認（スタッフ操作なし）

受渡確認はスタッフ向けアプリの機能ではない。提供台に設置された既存の自動受渡システム（電子パネル＋検知カメラ）が受け渡しを検知し、`HANDED_OVER` への遷移を自動で行う（§7.4）。スタッフ側で行うのは、完成した明細を提供台へ運び入れる操作（提供ボタン押下＋パネル指示位置への設置）のみで、Cognitoログインを伴う専用画面は不要（§7.4）。

## 6. 注文ステータス仕様

1 回の会計（注文）に複数商品が含まれ、かつ商品ごとに担当ゾーン（§7）が異なる場合がある。この場合、**商品ごとの製造進捗（明細ステータス）と、注文全体として顧客・スタッフへ提示するステータス（注文ステータス）を分けて管理する**。注文ステータスは明細ステータスから自動的に決まる導出値であり、直接書き換えることはない。（[Issue #11](https://github.com/Masaharu1223/AWS_OrderSystem/issues/11) で決定）

### 6.1 明細（FulfillmentLine）ステータス遷移図

明細は「注文に含まれる商品 1 品目」を指す（§7.2 のゾーン振り分けは明細単位で行われる）。各明細は、担当ゾーンごとに完全に独立して以下の遷移をたどる。他の明細の状態には一切影響されない。

```
WAITING ──→ PREPARING ──→ READY ──→ HANDED_OVER
   │
   └──→ CANCELLED（WAITING の間のみ可能）
```

| ステータス | 説明 | 遷移元 | 遷移先 |
| --- | --- | --- | --- |
| `WAITING` | 製造待ち（ゾーンのキューに並んでいる） | 注文確定 | PREPARING / CANCELLED |
| `PREPARING` | 製造中 | WAITING | READY |
| `READY` | 製造完了（この明細単体は完成済み） | PREPARING | HANDED_OVER |
| `HANDED_OVER` | 受渡済み（§6.3、検知カメラが持ち去りを検知して自動確定） | READY | — |
| `CANCELLED` | キャンセル済み | WAITING | — |

### 6.2 注文ステータス（導出値）

注文ステータスは、その注文に含まれる**全明細のステータスの組み合わせ**から一意に決まる。スタッフ・システムのどちらも注文ステータスへ直接書き込みは行わない。

| 注文ステータス | 決定条件（全明細に対する条件） |
| --- | --- |
| `STORE_ACCEPTED` | 全明細が `WAITING` |
| `PREPARING` | 1 明細以上が `PREPARING` または `READY`（かつ全明細 `READY` ではない） |
| `READY_PICKUP` | 全明細（`CANCELLED` を除く）が `READY` |
| `HANDED_OVER` | 全明細（`CANCELLED` を除く）が `HANDED_OVER` |
| `CANCELLED` | 全明細が `CANCELLED` |

`PENDING_PAYMENT`（決済待ち）と `PAYMENT_FAILED`（決済失敗）の 2 つは将来のアプリ内決済導入時の予約定義であり（§12.1）、明細がまだ作成されていない決済前の段階を表す。そのため上記の導出ルールの対象外とし、注文レコードへ直接書き込む値のまま維持する。

| ステータス | 説明 | 遷移元 | 遷移先 |
| --- | --- | --- | --- |
| `PENDING_PAYMENT` | 決済待ち（予約定義のみ、MVP未実装） | — | `STORE_ACCEPTED`（決済成功時に明細が作成される） |
| `PAYMENT_FAILED` | 決済失敗（予約定義のみ、MVP未実装） | `PENDING_PAYMENT` | — |

`READY_PICKUP`（受取可能）になるのは、**最も遅く完成したゾーンの明細が `READY` になった瞬間**である。顧客への「受取可能」通知（§9）は、この注文ステータスの変化を契機に送信する（明細単体の `READY` では送信しない）。

### 6.3 受け渡しは既存の自動受渡システムが検知する

明細が個別に `READY` になると、自動搬送レーンで提供台の電子パネルへ運ばれ、パネルが示す位置に設置される（§7.4）。**お客様が商品を取ると、パネル上部の検知カメラが持ち去りを検知し、当該明細を自動で `HANDED_OVER` に遷移させる。** スタッフによる受渡確認の操作は発生しない（§5.5）。注文ステータスの `HANDED_OVER`（§6.2）は、従来と同じ導出ルール（注文内の全明細〔`CANCELLED` を除く〕が `HANDED_OVER` になった時点で確定）のまま変わらない。複数ゾーンにまたがる注文でも、各明細のカメラ検知はゾーンごとに独立して行われ、全明細が検知された時点で注文全体が `HANDED_OVER` になる。

### 6.4 具体例

ラテ L（Zone A 担当）+ 紅茶 M（Zone D 担当）を 1 回の会計で注文した場合:

| 時刻 | Zone A の明細 | Zone D の明細 | 注文ステータス |
| --- | --- | --- | --- |
| 12:00 | WAITING | WAITING | STORE_ACCEPTED |
| 12:02 | PREPARING | WAITING | PREPARING |
| 12:04 | READY（ラテ完成） | PREPARING | PREPARING（紅茶待ち） |
| 12:07 | READY | READY（紅茶完成） | **READY_PICKUP**（ここで顧客へ通知） |
| 12:09 | HANDED_OVER（カメラが検知） | HANDED_OVER（カメラが検知） | HANDED_OVER |

ラテが 12:04 に完成しても、紅茶が揃う 12:07 まで顧客への「受取可能」通知は送らない。

## 7. マシンゾーン仕様

### 7.1 ゾーン定義


| ゾーン        | 対象商品                     | サイズ   | 物理マシン台数 |
| ---------- | ------------------------ | ----- | ------- |
| **Zone A** | エスプレッソ系                  | L のみ  | 1 台     |
| **Zone B** | エスプレッソ系                  | M のみ  | 1 台     |
| **Zone C** | エスプレッソ系                  | S のみ  | 1 台     |
| **Zone D** | エスプレッソ不使用（ティー系・フラペチーノ系等） | S / M | 1 台     |


### 7.2 振り分けロジック（MVP: ルールベース）

注文確定（DynamoDB への書き込み）を **DynamoDB Streams(INSERT) → EventBridge** が捕捉して `machine-router-fn` を起動する。同関数がドリンクカテゴリ + サイズから自動でゾーンを決定し、対応する **SQS FIFO キュー**（ゾーンごとに 1 本 = 計 4 本）へエンキューする。実際にキューを消費（consume）するのは `zone-consumer-fn`（プログラム）であり、バリスタ（人間）ではない。

```
エスプレッソ系 × L → Zone A
エスプレッソ系 × M → Zone B
エスプレッソ系 × S → Zone C
エスプレッソ不使用 × S/M → Zone D
```

### 7.2.1 なぜ SQS FIFO を採用するか（案A採用の根拠）

ゾーンへの振り分けは「DynamoDB に受付順（`createdAt`）を持たせるだけ（案B）」でもこの規模では実用上成立する。しかし本設計では以下の理由から **SQS FIFO を採用（案A）** する。


| 観点             | SQS FIFO 採用の効果                                                                         |
| -------------- | -------------------------------------------------------------------------------------- |
| **公平性（受付順保証）** | 同一ゾーン内で「先に注文した客が先に受け取れる」順序を AWS 基盤レベルで厳密に保証する。案B（`createdAt` 順）は同一ミリ秒の同着や時計ずれで順序が逆転しうる |
| **疎結合・バッファ**   | 注文書き込み（order-fn）と製造順の確定を分離できる。昼ピークの注文集中をキューで吸収する                                       |
| **重複排除**       | FIFO の重複排除（dedup）により二重エンキューを防ぐ（`Idempotency-Key` と二重の防御）                               |
| **将来拡張**       | 自動製造マシンを導入する際、キューをそのまま製造制御の入力として流用できる                                                  |


**トレードオフと結論**: `zone-consumer-fn` と 4 本のキューという部品増を伴う。案B なら部品ゼロで済むが、**受付順の厳密な公平性**と**将来の自動製造連携**を優先し、案A を採用する。

**消費モデルの注意**: SQS を取り出す（consume）のは `zone-consumer-fn`（プログラム）であり、バリスタ（人間）ではない。バリスタは DynamoDB を裏に持つスタッフ UI を見て操作する（§5）。SQS は「受付順の確定」に用い、一覧表示・検索は DynamoDB（GSI1）が担う、という役割分担とする。

### 7.3 キャンセル時のキュー除外

- キャンセルリクエスト受信時、DynamoDB の `status` を `CANCELLED` に更新する（条件: `status = STORE_ACCEPTED`）
- SQS からのメッセージ削除は行わない
- `zone-consumer-fn`（SQS コンシューマ）は処理前に DynamoDB の `status` を確認し、`CANCELLED` の場合はスキップする

### 7.4 店内動線（自動搬送レーン + 提供台）

ここまでの§7.1〜7.3はソフトウェア上の受付順管理（DynamoDB・SQS）の話であり、本節は店内の**物理的な**商品搬送動線を定義する。

各ゾーン（A〜D）で完成した商品は、スタッフの手渡しではなく**自動搬送レーン**（ベルトコンベア等の物理搬送設備）に乗せられ、店内中央の**提供台**まで自動で運ばれる。提供台には**既存の自動受渡システム**（電子パネル＋検知カメラ）を導入し、受け渡し工程そのものを自動化する。ゾーン担当（または搬入した者）が「提供ボタン」を押して電子パネルが指示する位置へ商品を設置すると、以降はスタッフの操作なしにパネル上部のカメラがお客様の持ち去りを検知し、受け渡しが完了する。専任の提供台担当ロールは置かない（§3.2・§5.1・§5.5）。

```mermaid
flowchart LR
    subgraph zones["ゾーン（製造機 + スタッフ用iPad）"]
        direction TB
        A["Zone A<br/>📱 iPad"]
        B["Zone B<br/>📱 iPad"]
        C["Zone C<br/>📱 iPad"]
        D["Zone D<br/>📱 iPad"]
    end
    Lane[自動搬送レーン<br/>(完成品を提供台まで搬送)]
    Panel["提供台<br/>電子パネル + 検知カメラ<br/>（既存の自動受渡システム）"]
    Customer["お客様<br/>商品を受け取る"]

    A -- 完成した商品（明細）--> Lane
    B -- 完成した商品（明細） --> Lane
    C -- 完成した商品（明細） --> Lane
    D -- 完成した商品（明細） --> Lane
    Lane -- 提供ボタン押下＋指示位置へ設置 --> Panel
    Panel -- 持ち去りをカメラが検知 --> Customer
```

**ソフトウェア上の状態遷移との対応**:

| 物理的な動き | ソフトウェア上の対応 |
| --- | --- |
| ゾーン担当が「完成」スワイプ → 商品を自動搬送レーンに乗せる | 明細ステータス `PREPARING → READY`（§5.3, §6.1） |
| 自動搬送レーンで提供台へ搬送中 | （搬送そのものはソフトウェアの管理対象外。搬送時間はETA計算に含めない、§8） |
| スタッフが「提供ボタン」を押し、電子パネル指示位置へ設置 | 既存の自動受渡システム側の操作。当システムの明細ステータスは変更しない |
| 全明細が提供台に揃う | 注文ステータスが導出で `READY_PICKUP` になる（§6.2）。顧客へ通知（§9） |
| お客様が商品を取る → カメラが持ち去りを検知 | 明細ステータス `READY → HANDED_OVER`（自動、§6.3）。全明細が揃うと注文ステータスも `HANDED_OVER` |

**留意点（MVPでの扱い）**:

- 自動搬送レーン・電子パネル・検知カメラはいずれも物理／外部システムであり、AWS側のソフトウェア費用（§10.5）とは別に導入・運用コストが発生する。本書のコスト試算（月額¥0〜数百円）はソフトウェア部分のみを対象とし、これらの費用は含まない
- 電子パネル・検知カメラは既存の自動受渡システム（外部システム）であり、当システムとの連携（提供ボタン操作時の明細登録、持ち去り検知時に当システムへ通知する方式など）の詳細は `architecture.md` 側で別途設計する
- 搬送レーンや検知カメラの故障・誤検知など物理的な異常系は、当システムでは検知できない。現場のゾーン担当が気づいた場合に声を掛け合う運用、または既存システム側の監視機能に依存する
- 複数ゾーンにまたがる注文は、電子パネル側で同一注文の商品をまとめて表示する運用を想定する。一部の商品だけ先に持ち去られた場合の扱いは既存システムの仕様に依存し、本書のスコープ外とする

## 8. ETA（製造完了予測）仕様

### 8.1 計測ポイント


| イベント       | タイミング                              |
| ---------- | ---------------------------------- |
| 製造開始時刻     | スタッフが「製造開始」スワイプ → `PREPARING` 遷移時刻 |
| 製造完了時刻     | スタッフが「完成」スワイプ → `READY_PICKUP` 遷移時刻       |
| エスプレッソ抽出時間 | 固定 25 秒（定数として扱い、学習対象から除外）          |


### 8.2 フェーズ 1（MVP）: ゾーン単位の指数移動平均

- 製造時間の実績をゾーン単位で蓄積
- 指数移動平均（EMA）で予測値を更新
- `GET /orders/{orderId}/queue-position` のレスポンスに `estimatedWaitMinutes` として返す

### 8.3 フェーズ 2（将来・データ 300〜500 件蓄積後）: Amazon Forecast

- ゾーン単位の細分化 → 商品単位の予測へ移行
- Amazon Forecast による時系列予測モデルの導入

## 9. リアルタイム通知仕様

### 9.1 WebSocket の動作


| 状態               | 動作                                                                                     |
| ---------------- | -------------------------------------------------------------------------------------- |
| フォアグラウンド（アプリ表示中） | WebSocket 接続を維持。ステータス変化を即時 push                                                        |
| バックグラウンド移行時      | OS がタブを停止 → WebSocket 自動切断（アプリは何もしない）                                                  |
| バックグラウンドからの復帰    | `Page Visibility API`（`visibilitychange` イベント）で検知 → REST API で最新状態を再取得 → WebSocket 再接続 |


### 9.2 WebSocket の活用範囲


| 利用者  | 通知内容                                       |
| ---- | ------------------------------------------ |
| 顧客   | 注文ステータスの変化（STORE_ACCEPTED → PREPARING → READY_PICKUP 等） |
| スタッフ | 新規注文の到着（新しいカードのリアルタイム追加）                   |


### 9.3 キュー位置のポーリング

- WebSocket では push しない（位置は注文状態ではなくキューの動的な状態）
- フォアグラウンド中のみ 30 秒間隔でポーリングを行う
- バックグラウンド時はポーリングを停止

## 10. 非機能要件

### 10.1 可用性

- AWS マネージドサービス（Lambda / DynamoDB / API Gateway）の AZ 冗長を活用
- SLA 目標: 99.9%（単一リージョン・単一 AZ 障害で継続稼働）

### 10.2 パフォーマンス

- API レスポンスタイム（P95）: 500ms 以内（Lambda コールドスタートを除く）
- WebSocket 通知遅延: 1 秒以内（DynamoDB Streams → EventBridge → push-fn → クライアント）

### 10.3 スケーラビリティ

- Lambda + DynamoDB オンデマンドにより、100 同時接続・昼ピーク時も自動スケール

### 10.4 セキュリティ


| 要件           | 対応                                        |
| ------------ | ----------------------------------------- |
| 通信経路の暗号化     | CloudFront / API Gateway で HTTPS 強制       |
| 保存データの暗号化    | DynamoDB デフォルト暗号化 / S3 SSE-S3             |
| 最小権限 IAM     | Lambda ごとに必要な DynamoDB 操作のみ許可（CDK grants） |
| 入力バリデーション    | サーバー側で enum 検証・個数上限・価格再計算                 |
| 二重注文防止       | `POST /orders` は `Idempotency-Key` に対応    |
| スロットリング      | API Gateway でバースト / レート上限を設定              |
| Staff API 認証 | Cognito JWT オーソライザー必須                     |


### 10.5 コスト

- アイドル時（夜間・週末）ほぼ ¥0
- 通常稼働時の目標: 月額 ¥1,000 未満


| サービス                 | 月額概算                     |
| -------------------- | ------------------------ |
| Lambda               | ¥0（無料枠 100 万 req/月 内）    |
| API Gateway HTTP API | ¥0〜数百円（$1.0 / 100 万 req） |
| DynamoDB オンデマンド      | 数百円                      |
| S3 + CloudFront      | 数十〜数百円                   |
| CloudWatch           | 数十円                      |
| **合計**               | **¥0〜数百円（$5 未満）**        |


### 10.6 可観測性

- 全 Lambda から CloudWatch Logs へ構造化 JSON ログを出力
- CloudWatch メトリクス / アラーム: 4xx / 5xx 率、Lambda エラー率、DynamoDB スロットル
- WebSocket 接続数・SQS キュー深度を監視

> **補足（インターフェース上の注記）**
>
> - `GET /menu/{productId}`: menu-fn は全メニュー（小規模・キャッシュ可能）を取得し、その中から該当商品を返す。パスに category を含めなくてよい。

> - **order-aggregator-fn**: 明細（LINE）の変化を **DynamoDB Streams から直接**（EventBridge を介さず）消費し、注文の導出ステータス（§6.2）を再計算する。技術的な詳細は `architecture.md` §9.8 を参照。

### 12.1 アプリ内決済 導入時のフロー（将来実装時の詳細）

MVP では「商品選択（メニュー・カート）→ 注文 → `STORE_ACCEPTED`（注文受付＝製造キューへの振り分け対象）」という、決済ステップを持たないフローで実装する。商用化にあたってアプリ内決済を導入する際は、`STORE_ACCEPTED` の手前に決済ステップを挟み込む形で、以下の手順を想定する。

1. `POST /orders` → 注文を `PENDING_PAYMENT`（決済待ち）状態で作成し、`orderId` を返す
2. クライアントを決済プロバイダ（Stripe 等）のチェックアウト画面へリダイレクトする
3. 決済完了 → Stripe Webhook を受信 → `payment-fn` が注文を `STORE_ACCEPTED` に更新し、WebSocket で push する
4. クライアントがチェックアウトから復帰 → WebSocket で状態変化を受信 → キュー位置の表示を開始する

- 決済失敗時は `PAYMENT_FAILED` に遷移し、カート内容は保持する（再決済できるようにするため）
- 「商品選択」自体は決済の有無に関わらず共通のフローであり、決済ステップだけを注文確定と製造キュー投入の間に後付けで挟み込める設計とする（§14.1 の Cognito 格上げと同様、`order-fn` 自体の構造は変えずに済む）

## 13. 用語定義


| 用語                      | 定義                                                      |
| ----------------------- | ------------------------------------------------------- |
| **ゾーン**                 | 物理マシン 1 台に対応した製造エリア（A / B / C / D）                      |
| **SQS FIFO**            | 各ゾーン専用の注文キュー。ゾーンごとに 1 本（計 4 本）                          |
| **ETA**                 | 製造完了予測時刻（Estimated Time of Arrival）                     |
| **sessionId**           | 未ログイン顧客を識別する UUID（ブラウザにローカル保持）                          |
| **orderId**             | 注文を一意に識別するシステム内部の UUID                                  |
| **注文番号**                | 顧客に提示する短い連番（例: #042）                                    |
| **variant**             | 商品の温度（hot/iced）+ サイズ（S/M/L）の組み合わせ                       |
| **ConnectionTable**     | WebSocket の connectionId と orderId を対応付ける DynamoDB テーブル |
| **Page Visibility API** | ブラウザが提供するタブのフォアグラウンド/バックグラウンド状態を検知する API                |
| **アンドゥバナー**             | スタッフのスワイプ操作後 3 秒間表示する取り消しバナー。3 秒経過で確定                   |
| **自動搬送レーン**             | 各ゾーンで完成した商品を提供台まで自動搬送する物理設備（§7.4）。ソフトウェアの管理対象外           |
| **提供台**                  | 自動搬送レーンの終点。既存の自動受渡システム（電子パネル＋検知カメラ）でお客様への受け渡しを検知する場所（§7.4） |
| **自動受渡システム**             | 提供台に設置された既存の外部システム（電子パネル＋検知カメラ）。「提供ボタン」押下で商品の設置位置を指示し、お客様の持ち去りをカメラで検知して `HANDED_OVER` を自動確定する（§5.5, §6.3, §7.4）。専任スタッフのログインロールは持たない |

## 14. 主要な設計判断の経緯

本セクションは確定仕様の「なぜ」を記録し、将来の再検討時に判断基準を参照できるようにするためのもの。旧プランファイル（`nifty-wishing-origami.md` / `todo-snug-scott.md`）に記録されていた議論の要点をここに統合する。

### 14.1 Cognito 認証の位置づけ変更

- 初期設計ではスコープ外（将来の「特別メニュー」機能向けの布石）だった
- スタッフ用画面（ゾーン選択・注文管理）へのアクセス制御が別途必要と判明し、**MVP必須に格上げ**した
- 個人別アカウントではなく共有アカウント1つ（最大4台タブレット同時ログイン）とし、運用をシンプルに保った

### 14.2 リアルタイム通知に WebSocket を採用（ポーリングではなく）

- Well-Architected の「パフォーマンス効率」の観点。ポーリングよりリクエスト数が減り、状態変化時のみ通信するため無駄な DynamoDB Read が発生しない
- バックグラウンドでの WebSocket 維持はブラウザ仕様上不可能なため、`Page Visibility API` による復帰時の再接続方式を採用
- Web Push（FCM経由）は通知許可のUXコストがあるため、まずはフォアグラウンド前提の設計とし将来拡張に回した

### 14.3 注文キャンセル方式の比較と選定


| 方式                                   | 速度  | コスト          | 複雑さ | 判定                                 |
| ------------------------------------ | --- | ------------ | --- | ---------------------------------- |
| A. DynamoDB 条件付き Update + 消費側の冪等チェック | 高   | 低（追加コストほぼゼロ） | 低   | **採用**                             |
| B. SQS メッセージ削除                       | 中   | 中            | 中   | 見送り（`ReceiptHandle` の保存コストが発生するため） |
| C. DynamoDB Streams でキャンセルイベントを伝播    | 低   | 低            | 高   | 見送り（複雑さが見合わない）                     |


### 14.4 マシンルーティング: ルールベース先行、Bedrock は段階導入

- MVP は if / switch の**ルールベース**で十分と判断（実装・テストが容易）
- Bedrock は「キュー深度に応じた動的振り分け」「処理時間の異常検知」など複雑な最適化が必要になった段階（フェーズ2）で初めて価値が出るため、データが溜まってから導入判断する

### 14.5 ETA 予測: EMA 先行、Amazon Forecast は段階導入（Bedrock は不採用）

- ETA 予測は時系列予測タスクであるため、汎用 LLM の Bedrock ではなく専用の時系列予測サービスである **Amazon Forecast** を採用先として選定。
- データが 300〜500 件蓄積されるまでは、フェーズ1としてゾーン単位（A, B, C, Dの４つ）の指数移動平均（EMA）で代替する段階的導入方針とした。

### 14.6 EventBridge と SQS の使い分け方針

- **SQS FIFO**: 「送り主1つ→受け取り先1つへ、順番を守って渡したいだけ」の処理に使う。ゾーンごとの製造キュー（1台のマシンへ注文を受付順に渡す）がこれに当たる
- **EventBridge**: 「1つの出来事（例: 注文が入った、ステータスが変わった）を、複数の異なる担当処理に同時に知らせたい」場合に使う。「注文が入った」という1つの出来事をきっかけに、マシンへの振り分け・顧客への通知・（将来の）売上集計・（将来の）ポイント付与など、複数の処理を同時に走らせられる
- この使い分けにより、将来新しい機能（通知・ポイント付与・売上集計・外部POS連携など）を追加したくなっても、注文を受け付ける `order-fn` 自体のコードは変更せず、EventBridge側に新しい「知らせる相手」を追加登録するだけで対応できる（§7.2, §7.2.1 参照）

### 14.10 複数ゾーン注文のステータス確定方式（明細単位モデルの採用）

- 当初の設計（v1.1）では、注文ステータス（`STORE_ACCEPTED`/`PREPARING`/`READY_PICKUP` 等）を注文レコード 1 件に対して 1 つだけ持たせていた。しかし §7.2 の通りゾーン振り分けは商品（明細）ごとに行われるため、1 注文が複数ゾーンにまたがると、後から処理したゾーンの状態で先の状態が上書きされ、一部のゾーンにしか商品が表示されない・一部完成のみで受取可能と誤通知される、といった問題が発生することが判明した（[Issue #6](https://github.com/Masaharu1223/AWS_OrderSystem/issues/6) で指摘、後に [Issue #11](https://github.com/Masaharu1223/AWS_OrderSystem/issues/11) の決定に伴い Issue #6 は削除）
- 対応として、商品（明細）ごとに独立したステータスを持つモデルへ変更した（§6）。注文全体のステータスは、全明細の状態から導出する値とし、直接の書き込み対象から外した
- 受取可能（`READY_PICKUP`）は「最も遅いゾーンの明細が完成した瞬間」に確定する。全ゾーンが完了するまで顧客への完成通知を待つことで、駆け付けた顧客が一部の商品しか受け取れない事態を防ぐ
- 受け渡しは、渡し忘れ事故を防ぐため商品単位ではなく注文単位で一括確定する（§6.3）
- キャンセル可否の境界は「全明細が製造未着手（`WAITING`）かどうか」に変更した（§4.5）。従来の「注文ステータスが `STORE_ACCEPTED` かどうか」は、新モデルでは全明細 `WAITING` の場合の導出値と一致するため、実質的な挙動は変わらない
- 本節に付随する技術的な検討事項（明細ごとのデータモデル詳細、同一商品複数個の明細分割方針、ゾーン別一覧のアクセスパターン再設計等）は、architecture.md 改訂時に確定する（[Issue #11](https://github.com/Masaharu1223/AWS_OrderSystem/issues/11) の検討事項を参照）

### 14.11 提供台の受渡確認を自動検知方式へ変更（提供台担当ロールの廃止）

- 当初案（§7.4 初版）では、提供台に「提供台担当」という専任のログインロールを置き、スタッフが自動搬送レーンで届いた商品と注文内容を目視照合したうえで、注文単位の「受渡確認」ボタンを手動で押す設計だった
- 見直しの結果、既存の自動受渡システム（電子パネル＋検知カメラ）を採用する方式に変更した。スタッフ（搬入した者）が「提供ボタン」を押して電子パネル指示位置へ商品を設置し、以降はお客様の持ち去りをカメラが検知して `HANDED_OVER` を自動確定する（§5.5, §6.3, §7.4）
- この変更に伴い、スタッフ選択画面（§5.1）は A/B/C/D の4択に戻り、提供台担当というログインロール・専用画面は廃止した（§3.2）
- `READY_PICKUP` への導出ルール（§6.2、全明細 `READY` で確定）は変更していない。変わったのは `READY → HANDED_OVER` の遷移トリガーのみで、「スタッフの手動一括確認」から「検知カメラによる自動検知（明細ごとに独立、全明細検知で注文単位のHANDED_OVERへ導出）」に置き換わった
- 電子パネル・検知カメラとの連携方式（提供ボタン操作時のデータ登録、持ち去り検知時の当システムへの通知方法など）は既存の外部システムとの統合であり、詳細は `architecture.md` 側で別途設計する
